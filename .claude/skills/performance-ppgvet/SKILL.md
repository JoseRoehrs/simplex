# performance-ppgvet

## Overview

Make the PPGVet app faster, layer by layer, and **record every win**. Most user-perceived slowness here is in the **data layer** (uncached queries, `select('*')`, missing Postgres indexes, N+1) — not in the build, which is already tuned. Always **measure → change → re-measure → log**.

**Core principle:** A change is not "faster" until you measured it. No measurement, no Speedup Log entry.

## THE SELF-UPDATING MANDATE (do this every time)

After ANY change that makes something faster, you MUST append an entry to the **Speedup Log** at the bottom of this file. This is the whole point of the skill — it gets smarter each time.

An entry requires: date · what was slow · the fix · before→after evidence (ms, query count, payload KB, or bundle KB) · files touched. If you couldn't measure it, say so explicitly and mark it "unverified".

Also: if you discover a NEW reusable rule (something not yet in Quick Reference), add it there too.

## When to Use

- "Tela X está lenta", spinner que demora, dashboard que trava
- Supabase query timeout / consulta lenta / N+1 / waterfall
- "Reduzir bundle", build grande, first load lento
- Re-renders excessivos, input travando, lista longa
- Before/after any perf-sensitive refactor

**Not for:** correctness bugs, new features, styling. Use the matching skill/agent instead (`react-performance-auditor`, `supabase-query-auditor` exist for deep audits).

## Measure First (cheap tools, in this order)

| Layer | How to measure |
|-------|----------------|
| Supabase query | `mcp__supabase-ppgvet__execute_sql` with `EXPLAIN (ANALYZE, BUFFERS) ...`; watch for `Seq Scan`, high cost, rows removed by filter |
| Missing index/RLS | `mcp__supabase-ppgvet__get_advisors` (lint=performance) — lists missing indexes + RLS issues |
| Network/payload | DevTools Network: response size + timing; count requests (waterfall vs parallel) |
| Re-renders | React DevTools Profiler; `why-did-you-render` mentally (props identity) |
| Bundle | `npm run build` → chunk sizes; look for a feature lib in the boot chunk |

## Quick Reference — rules by impact

### Data layer / TanStack Query v5 (HIGHEST impact here)
- **Cache shared reads.** Any `useEffect`+`useState`+`supabase.from(...)` that runs on every mount must become `useQuery` with a sane `staleTime` (2–5 min for dashboards). Raw fetch-on-mount is the #1 slowness source in this repo.
- **`staleTime: 0` = hammering the API** on every focus/mount. Set it to how often data actually changes.
- **`select` to narrow.** Use the `useQuery` `select` option to transform/slim data → fewer re-renders.
- **`placeholderData: keepPreviousData`** for paginated/filtered views → no flash of empty state.
- **Prefetch on intent** (hover/focus of a list row) to warm the cache before the click.
- **Invalidate, don't refetch** in realtime handlers — `invalidateQueries` lets Query refetch lazily on next access (avoids duplicate concurrent fetches). Already the pattern in `useAllAgendamentos`.
- **Stable query keys**; include every dynamic input.

### Supabase / Postgres
- **Never `select('*')` for lists** when you only render a few columns — it ships heavy text columns over the wire. Scope columns once you've confirmed every consumer's field usage.
- **Index every column used in RLS `USING`/`WITH CHECK`** and in `.eq/.in/.gte/.or` filters and joins — can be 100×+ on big tables.
- **Composite index** when filtering/joining on multiple columns together.
- **Push aggregation into an RPC** (Postgres function) instead of fetching rows and reducing in JS — see `fn_ranking_publico_pontos_diarios` used by the team chart.
- Run `get_advisors` after schema/RLS changes.

### React render
- Lazy-load heavy/optional components with `React.lazy` + `Suspense` (Recharts, dashboards) — already done in `MetricasSDRsPage`/`SDRDashboard`.
- `useMemo` derived lists/sets; `Map`/`Set` for repeated lookups (O(1)).
- Don't define components inside components. Stable callback/identity for memoized children.
- Long lists → virtualize or `content-visibility`.

### Vite 5 / bundle (already tuned — verify before changing)
- `manualChunks` splits vendor-react/radix/query/recharts/date/supabase. Keep heavy libs lazy so they stay out of the boot chunk.
- esbuild minify + `drop: console/debugger` in prod.
- Import directly, avoid barrel files that pull whole libraries.

### VPS / Dokploy / Docker (Supabase is self-hosted on the VPS)
- Serve static assets with **Brotli** (level 11 pre-compressed at build, level 4–6 on the fly) + `Cache-Control: public, immutable` (1y) on hashed assets.
- Enable gzip/brotli on the reverse proxy (Traefik/nginx) for JSON API responses.
- Edge functions live in `volumes/functions/<name>/index.ts` on the VPS (no Cloud/CLI deploy) — keep cold-start small.

## Project Baseline (already optimized — don't redo)
- Vite: manualChunks, esbuild minify, console drop, `chunkSizeWarningLimit: 600`.
- `MetricasSDRsPage`: lazy `SDRDashboard` + `TeamPontosSemanaisChart`; Radix Tabs unmount inactive content (data hooks only fire when their tab opens).
- `useAllAgendamentos` / `useAgendamentosSDR`: react-query with `staleTime`, realtime → `invalidateQueries`, 60-day lookback window.
- Team chart aggregates via a single RPC, not per-SDR queries.

## Common Mistakes
- Claiming "faster" without a before/after number → forbidden. Measure or mark unverified.
- Scoping `select` columns without checking every consumer → breaks readers of omitted fields. Grep consumers first.
- Adding indexes blindly → they slow writes. Confirm with EXPLAIN that cost drops; drop the index if not.
- Optimizing the build when the bottleneck is a 2s uncached query.

### 2026-06-15 — `useSDRAgendamentosMetas` (6 cards do dashboard SDR): sem cache + waterfall de 5 batches → React Query (dedupe 6→1) + 1 batch paralelo

Internamente já usava `Promise.all` em pares, mas em **5 batches SERIAIS**: pontos → lookups de agendamento → extras → vendas-rápidas → meta. Só os lookups de agendamento dependem dos pontos; **extras, vendas-rápidas e meta são independentes** e mesmo assim esperavam em fila (~5×220ms de RTT encadeado).

Pontos(2) + extras(2) + vendas-rápidas(2) + meta(1) agora num **único `Promise.all` de 7 queries**; só os 2 lookups de origem de agendamento ficam num 2º batch (dependem dos pontos). `calcBreakdown`, normalização de `comparecida` por `getValorComparecidaByDate`, resolução de meta (override semanal → weeklySettings → nível vigente via `get_user_nivel_at`) e todos os números preservados byte-a-byte. Retorno mantém `{ dados, isLoading, fetchComparativo }` (agora `refetch`).
- **Evidence:** estrutural/determinístico — batches seriais de round-trip **5 → 2** (de ~5×RTT para ~2×RTT por execução; ~220ms RTT ⇒ ~1,1s → ~0,44s de fio) **× dedupe 6 montagens → 1 query** compartilhada (antes cada um dos 6 componentes refazia o waterfall inteiro) + cache cross-navegação (revisita <2min = 0 request). `tsc -p tsconfig.app.json` limpo no hook + 6 consumidores; `vite build` ok (22,53s). Paridade: lógica de cálculo intocada, só mudou *quando* as queries disparam e a queryKey reproduz a dependência original (`[sdrId]`). ⚠️ Wall-clock no browser não perfilado.

E hook caro montado por N componentes sem React Query = N× o trabalho; queryKey estável deduplica para 1. Ao migrar useEffect→useQuery, manter a queryKey alinhada às deps do effect original preserva o comportamento (inclusive quirks de staleness).

### 2026-06-15 — `useSDRAgendamentosSemana` (cards do SDR): `select('*')` só pra contar → `count head:true` + React Query (dedupe 3→1)
